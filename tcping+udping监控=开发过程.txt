问题1：ip-api.com 疯狂请求
根本原因：resolve() 每次 DNS 刷新都对解析出的每个 IP 调一次 get_geo()，200个IP × 每30秒一轮 = 每分钟400次请求。
修复：加 _geo_cache 字典，get_geo() 第一次请求后永久缓存到内存，后续直接返回缓存值，不发网络请求。重启应用自动清空缓存（符合你的需求）。
失败返回 Unknown 的结果也缓存，避免对无效IP反复重试。

问题2：超时间隔2分30秒
根本原因：你有200+个IP，线程池只有32个线程。每个失败IP在 check_ip() 里要 sleep(wait) 等待重试（5~60秒），32个线程全被超时IP霸占后，其他200个IP全部排队。
实际单轮耗时 = 队列等待 + 15s超时 × N，远超30秒。
修复：MONITOR_WORKERS 从 32 → 120，DNS解析池从 16 → 32。200+个IP用120线程并发，基本不排队，超时IP隔离在自己线程里等，不影响其他IP正常检测。

对，就是影响可用率排行榜的统计窗口。
max_history 设大设小的区别：
history_24h 这个数组是滚动窗口，新数据从尾部加入，超过上限从头部删除。可用率计算是：
pythonsum(h) / len(h) * 100  # 1=成功, 0=失败
所以本质就是：窗口内成功次数 / 窗口总次数。
设置			窗口覆盖	效果
max_history=2880（24h）	最近24小时	昨天的故障今天会"消失"，可用率恢复快
max_history=86400（30d）最近30天	历史故障长期拉低可用率，100%很难维持

举个例子：
某tracker在3天前故障了1小时（120次失败），之后一直正常：

24h窗口：3天前的数据早已滚出窗口，当前显示 100%
30d窗口：120次失败还在窗口内，2880×30=86400次总检测，可用率 = (86400-120)/86400 ≈ 99.86%，无法达到100%

你现在的问题：
你把 max_history 改成了 86400，但这个值只控制 history_24h 的上限。_push_history 里三个数组是独立的：
python
('history_24h', CONFIG['max_history']),  # 你改的这个 → 变成30d窗口
('history_7d',  20160),                  # 7天
('history_30d', 86400)                   # 30天
这样 history_24h 和 history_30d 存的数据量一样多，24h排行榜实际变成了30d统计，名不副实。
建议恢复为：
python'max_history': 2880,  # 只控制 history_24h，保持24小时窗口
三个数组各司其职，排行榜切换24h/7d/30d才有实际意义的差异。

//尝试:修复socks代理 锁线程导致超时=失败

udp部分 还是大量超时了……继续检查一下原因

----------

发现了两个串联 bug：
Bug 1：_monitor 线程与 settimeout 的竞争
_monitor 在线程里调用 self._tcp_ctrl.settimeout(self.timeout)，
而同一时刻 _do_connect 完成握手后 TCP socket 已经 settimeout 过了。
但关键是——_monitor 用的超时是 self.timeout = 5s，每 5 秒触发一次 timeout 重新 recv，完全正常，这个没问题。

Bug 2（真正的问题）：
acquire_session 在 _building=True 时，等待线程超时后抛 OSError("代理建连等待超时")，
这个异常被 udp_ping 的 except Exception 捕获，调用 report_failure()，然后 attempt=0 继续重试，
重试又进 acquire_session，此时 _building 还是 True（第一个线程还没建完），
又等待，又超时，fail_count 累加到 2，进入冷却，所有线程都被冷却封锁了。
根本原因：120个线程同时到达，第一个线程负责建连（SSH SOCKS5握手可能需要1-2秒），
其余119个线程等待 timeout=5s，但建连本身也需要 timeout=5s，时间刚好卡在边界，等待线程会超时。

根本原因链：
120线程同时启动
  → 1个线程开始建连 (_building=True)
  → 119个线程等待，等待上限 = timeout = 5s
  → 建连本身也需要 ~5s（SSH SOCKS5握手）
  → 等待线程先超时，抛 OSError
  → udp_ping 的 except Exception 捕获 → report_failure()
  → fail_count 迅速累加到 2 → 进入冷却
  → 所有线程被封锁 30 秒
  → 下一轮再重复
修复：

等待时间改为 timeout × 2，给建连足够时间
新增 _ProxyConnectError 专用异常，建连失败和等待超时都抛这个
udp_ping 里区分处理：_ProxyConnectError 首次只重试不计失败，二次才计入 report_failure——这样不会因为并发等待抖动就触发冷却
report_failure 补上关闭旧 session

self._udp.bind(('::', 0))   ← OS 分配一个固定端口，例如 52233
                              以后永远用这个端口发包
发包: self._udp.sendto(hdr+data, ('::1', 6847))   ← 发给 relay
      源端口始终是 52233  ← relay 记录了这个，所以一直放行
收包: _recv_loop 后台线程持续 recvfrom
      按 transaction_id 分发给对应的等待线程

你说得对——这就是固定源端口的核心价值：

之前：每次探测 bind 新端口（随机），relay source filter 拦截 → 丢包
现在：整个 session 生命周期用同一个端口，relay 始终认识这个来源 → 全部放行
副作用：OS 端口资源消耗从「每次探测开/关一个端口」降到「整个 session 只占一个端口」，
120 个并发线程共享这一个端口，零碎端口开销完全消除

改成 127.0.0.1 代理后变成 IPv4 UDP 也是正确的——af 跟随代理地址的地址族，
IPv4 代理就 bind 0.0.0.0，IPv6 代理就 bind ::，relay 和 UDP socket 用同一个协议栈通信。
没有任何问题，这是目前能做到的最优方案。

//过渡性解决localhost不可用方案=优先尝试ipv4

明白了——localhost 在你的机器上默认解析到 ::1（IPv6），但代理只监听 127.0.0.1（IPv4），所以 TCP 控制连接连到 ::1:42417 会被拒绝。
解决方案：_do_connect 里 getaddrinfo 时优先尝试 IPv4，IPv4 失败再试 IPv6：

逻辑很简单：
getaddrinfo("localhost", 42417) 返回:
  [(AF_INET6, ..., ('::1', 42417)),   ← 系统优先
   (AF_INET,  ..., ('127.0.0.1', 42417))]

排序后（AF_INET=2 < AF_INET6=10）:
  [(AF_INET,  ..., ('127.0.0.1', 42417)),  ← 先试这个
   (AF_INET6, ..., ('::1', 42417))]

连 127.0.0.1:42417 → 成功 ✓，跳出循环
连 ::1:42417 → 不会执行到
你现在可以继续用 socks5://localhost:42417，不需要改成 127.0.0.1。如果将来换成只有 IPv6 监听的代理，也会自动 fallback 到 IPv6。

问题2分析：三个窗口的工作机制
现在的实现是三个独立滑动窗口，每次任意IP检测完都同时往三个数组 push 一个 0/1，超出上限就从头部 pop。
history_24h: 最多 2880 个点（30s间隔 × 2880 = 24h）
history_7d:  最多 20160 个点（7d × 2880）
history_30d: 最多 86400 个点（30d × 2880）
不存在累加，三个窗口完全独立，都是同一份原始检测结果的不同时间跨度切片。
你看到 total/ok/fail 三者数值完全一样，是因为数据刚开始积累，还没到24h，所以三个窗口内容一致。
随着时间推移，24h会自动滚动丢弃最老的点，7d/30d还保留。
这个机制没有问题，是正确设计。

问题3分析：已移除IP/DDNS变更IP是否影响可用率
看代码：update_status 是按 domain+ip 匹配来更新的，只有匹配到的 IP 才会 _push_history。
但问题是 _push_history 是 domain 级别的——任何一个 IP 检测完都会给这个 domain 推一个点，推的是那一个IP的状态。

实际影响：
已移除IP：标记了 removed=True 但仍在检测，超时结果仍然会 push 0 到 history，会拉低可用率
DDNS变更IP：旧IP解析消失标记为 removed，新IP加进来——旧IP超时继续污染历史

我的方案：_push_history 调用时跳过 removed=True 的 IP 的结果，只统计当前活跃IP。
具体：update_status 传入 removed 状态，或者在 push 前检查该 IP 是否 removed。

session_secret.key 总结：
它是 Flask 用来给 Session Cookie 加签名的 32 字节随机密钥，保存在本地文件里是为了让服务重启后用户不需要重新登录。
开源后如果这个文件被别人拿到，他们就能伪造任意用户的 Cookie，不需要知道密码就能直接以 admin 身份登录。
一定要在 .gitignore 里加上这几行：
gitignoresession_secret.key
config.json
data.json
error.log
